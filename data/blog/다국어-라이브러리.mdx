# 📝새로 배운 개념

## 개념

## **1. i18next**

### **개요**

i18next는 강력하고 유연한 다국어 지원 라이브러리입니다. 다양한 플러그인과 확장을 지원하며, 백엔드 번역 저장소와의 통합도 용이합니다.

### **주요 기능**

- 다양한 백엔드 지원 (XHR, 파일 시스템 등)
- 플러그인 기반 아키텍처
- 다국어 처리 및 플러럴라이제이션 지원
- 사용이 간편한 React 바인딩 (react-i18next)

### **설치 및 사용 예제**

```
npm install i18next react-i18next
```

```jsx
import React from 'react';
import ReactDOM from 'react-dom';
import App from './App';
import i18n from 'i18next';
import { I18nextProvider } from 'react-i18next';
import { initReactI18next } from 'react-i18next';

i18n
  .use(initReactI18next)
  .init({
    resources: {
      en: {
        translation: {
          "welcome": "Welcome to React and react-i18next"
        }
      },
      ko: {
        translation: {
          "welcome": "React와 react-i18next에 오신 것을 환영합니다"
        }
      }
    },
    lng: "en",
    fallbackLng: "en",
    interpolation: {
      escapeValue: false
    }
  });

ReactDOM.render(
  <I18nextProvider i18n={i18n}>
    <App />
  </I18nextProvider>,
  document.getElementById('root')
);`

`import React from 'react';
import { useTranslation } from 'react-i18next';

const App = () => {
  const { t } = useTranslation();

  return (
    <div>
      <h1>{t('welcome')}</h1>
    </div>
  );
};

export default App;
```

### **장점**

- 다양한 기능과 플러그인 지원
- 유연한 설정
- 강력한 플러럴라이제이션 기능

### **단점**

- 설정이 복잡할 수 있음
- 상대적으로 큰 번들 크기

### **공식 문서**

[**i18next 공식 문서**](https://www.i18next.com/)

## **2. react-intl**

### **개요**

react-intl은 국제화 표준을 준수하는 React용 라이브러리입니다. FormatJS 프로젝트의 일부로, ICU 메시지 형식을 지원하여 다양한 언어와 문화권에 맞춘 포맷팅을 제공합니다.

### **주요 기능**

- ICU 메시지 형식 지원
- 날짜, 시간, 숫자, 통화 포맷팅
- 컴포넌트 기반의 간단한 사용법

### **설치 및 사용 예제**

```
npm install react-intl
```

```jsx
import React from 'react';
import ReactDOM from 'react-dom';
import { IntlProvider } from 'react-intl';
import App from './App';

const messages = {
  en: { "welcome": "Welcome to React and react-intl" },
  ko: { "welcome": "React와 react-intl에 오신 것을 환영합니다" }
};

const language = navigator.language.split(/[-_]/)[0];

ReactDOM.render(
  <IntlProvider locale={language} messages={messages[language]}>
    <App />
  </IntlProvider>,
  document.getElementById('root')
);`

`import React from 'react';
import { useIntl } from 'react-intl';

const App = () => {
  const intl = useIntl();

  return (
    <div>
      <h1>{intl.formatMessage({ id: 'welcome' })}</h1>
    </div>
  );
};

export default App;
```

### **장점**

- 표준화된 국제화 지원
- 강력한 포맷팅 기능
- 비교적 간단한 설정

### **단점**

- ICU 메시지 형식 학습 필요
- 상대적으로 적은 커뮤니티 지원

### **공식 문서**

[**react-intl 공식 문서**](https://formatjs.io/docs/react-intl)

## **3. polyglot.js**

### **개요**

polyglot.js는 간단한 번역 라이브러리로, 주로 작은 프로젝트나 기본적인 다국어 지원이 필요한 경우에 적합합니다.

### **주요 기능**

- 간단한 사용법
- 작은 번들 크기
- 기본적인 플러럴라이제이션 지원

### **설치 및 사용 예제**

```
npm install node-polyglot
```

```jsx
import React from 'react';
import ReactDOM from 'react-dom';
import Polyglot from 'node-polyglot';
import App from './App';

const polyglot = new Polyglot({
  phrases: {
    "en": { "welcome": "Welcome to React and polyglot" },
    "ko": { "welcome": "React와 polyglot에 오신 것을 환영합니다" }
  },
  locale: "en"
});

ReactDOM.render(
  <App polyglot={polyglot} />,
  document.getElementById('root')
);`

`import React from 'react';

const App = ({ polyglot }) => {
  return (
    <div>
      <h1>{polyglot.t('welcome')}</h1>
    </div>
  );
};

export default App;
```

### **장점**

- 매우 간단하고 가벼움
- 설정이 쉬움

### **단점**

- 제한된 기능
- 대규모 프로젝트에 부적합

### **공식 문서**

[**polyglot.js 공식 문서**](https://airbnb.io/polyglot.js/)

## **i18n (internationalization) 및 i10n (localization)**

### **i18n (Internationalization)**

국제화는 소프트웨어가 다양한 언어와 문화권을 지원할 수 있도록 설계하는 과정을 의미합니다. 여기에는 텍스트 문자열을 외부 파일로 추출하고, 다국어 번역을 지원하며, 다양한 시간대, 통화, 날짜 형식을 처리하는 기능이 포함됩니다.

### **i10n (Localization)**

현지화는 국제화된 소프트웨어를 특정 언어와 문화권에 맞추는 과정을 의미합니다. 여기에는 텍스트 번역, 지역 특화된 콘텐츠 추가, 지역 규범에 맞춘 포맷팅 조정이 포함됩니다.

### **주요 차이점**

- **국제화 (i18n)**: 소프트웨어가 다국어와 다양한 문화권을 지원할 수 있도록 준비하는 과정
- **현지화 (i10n)**: 국제화된 소프트웨어를 특정 언어와 문화권에 맞추는 과정

# 다국어 지원 추가 내용

**다국어 지원의 중요성**

다국어 지원은 글로벌 서비스를 운영하는 데 필수적인 요소입니다. 특히 사용자 경험(UX)을 높이기 위해, 사용자들이 자신이 편리하게 이해할 수 있는 언어로 서비스를 제공하는 것은 필수적입니다. 그렇다면, 다국어 지원을 어떻게 구현할 수 있을까요? 여기서 우리는 props drilling 문제를 최소화하면서 다국어 지원을 할 수 있는 몇 가지 방법에 대해 논의하고자 합니다.

### **1. Higher-Order Component (HOC) 방식**

**HOC란?**

Higher-Order Component는 하나의 컴포넌트를 인자로 받아 새로운 기능을 추가한 컴포넌트를 반환하는 함수형 패턴입니다. 이 패턴을 사용하면 공통된 로직을 여러 컴포넌트에서 재사용할 수 있습니다. 다국어 지원을 위한 로직을 HOC에 구현하여 각 페이지나 컴포넌트에서 동일하게 사용할 수 있습니다.

**구현 예시**

```jsx
const withTranslation = (WrappedComponent) => {
  return function TranslatedComponent(props) {
    const lang = useLanguage() // 현재 선택된 언어 정보를 가져옴
    const translations = getTranslations(lang) // 언어에 맞는 번역 데이터를 가져옴

    return <WrappedComponent {...props} translations={translations} />
  }
}
```

**장점**

- 코드 재사용성을 높여 중복 코드를 줄일 수 있습니다.
- 다국어 관련 로직을 한 곳에 모아 관리할 수 있어 유지보수에 용이합니다.

**단점**

- 컴포넌트가 중첩될 경우 복잡도가 증가할 수 있습니다.
- 함수 호출이 많아져 성능에 영향을 줄 수 있습니다.

### **2. 모듈화된 상태 관리 방식**

**모듈화된 상태 관리란?**

모든 컴포넌트가 중앙에서 관리되는 전역 상태를 사용하지 않고, 각 컴포넌트나 모듈 내에서 독립적으로 상태를 관리하는 방식입니다. 각 컴포넌트는 독립적으로 언어 상태를 관리하며, 언어 변경 시 상위 컴포넌트에서 언어를 설정해주는 함수를 받아 사용할 수 있습니다.

**구현 예시**

```jsx
function PageComponent() {
  const [lang, setLang] = useState('en')

  return (
    <div>
      <LanguageSelector onChange={(newLang) => setLang(newLang)} />
      <Content lang={lang} />
    </div>
  )
}
```

**장점**

- 상태 관리가 개별 컴포넌트에 국한되므로 코드가 깔끔하고 직관적입니다.
- 각 모듈이나 컴포넌트가 독립적으로 관리되기 때문에 props drilling을 어느 정도 방지할 수 있습니다.

**단점**

- 언어 변경과 같은 전역 상태를 컴포넌트 간에 공유하는 것이 어려울 수 있습니다.
- 언어 변경 로직을 여러 곳에서 중복 관리하게 되면 유지보수가 어려워질 수 있습니다.

### **3. URL 라우팅을 이용한 언어 설정**

**URL 라우팅이란?**

언어 설정을 URL의 일부로 포함시켜 각 언어별로 다른 경로를 가지게 만드는 방식입니다. 예를 들어, `/en`, `/ko`와 같은 경로를 사용하여 해당 경로에 맞는 언어 데이터를 제공할 수 있습니다. 서버 측에서 이 경로를 통해 언어를 결정하고, 클라이언트에 해당 언어 데이터를 전달하는 방식입니다.

**구현 예시**

```jsx
import { useRouter } from 'next/router'

function PageComponent() {
  const router = useRouter()
  const { lang } = router.query // URL에서 언어 정보를 가져옴

  return (
    <div>
      <h1>{lang === 'ko' ? '안녕하세요' : 'Hello'}</h1>
    </div>
  )
}
```

**장점**

- URL을 통해 언어를 명시적으로 관리하므로 props drilling이 발생하지 않습니다.
- 사용자가 URL을 통해 직접 언어를 변경할 수 있으므로 사용자 경험이 향상됩니다.
- 서버 사이드에서 언어를 처리할 수 있어 성능 향상에 기여할 수 있습니다.

**단점**

- URL 구조가 복잡해질 수 있으며, 라우팅 설정이 추가로 필요합니다.
- SSR(서버사이드 렌더링)을 사용하는 경우에는 초기 렌더링 시 서버에서 언어 설정을 처리해야 하므로 추가적인 서버 로직이 필요합니다.
