# MPA

### MPA(Multi Page Application)에 대한 개요

- \*MPA(Multi Page Application)\*\*는 전통적인 웹 애플리케이션 구조로, 각 페이지가 독립적인 HTML 파일로 존재하며, 사용자가 다른 페이지로 이동할 때마다 서버로부터 새로운 HTML 파일을 요청하고 전체 페이지를 새로고침하는 방식입니다. 웹사이트의 다양한 페이지들 사이에서 이동할 때마다 브라우저는 각기 다른 URL에 대해 서버에 요청을 보내고, 서버는 요청에 맞는 전체 HTML 파일을 반환합니다.

이 방식은 초창기 웹 애플리케이션에서 많이 사용되었으며, 페이지 간 전환 시마다 서버와의 통신을 통해 새로운 페이지를 제공하는 구조입니다. 각 페이지는 독립적으로 구성되어 있으며, 각 페이지마다 HTML, CSS, JavaScript 등의 리소스가 필요합니다.

### MPA의 장점

1. **SEO 친화적**

각 페이지가 별도의 URL을 가지기 때문에 검색 엔진에서 크롤링하기 용이합니다. 따라서 SEO(Search Engine Optimization)에 강점을 가지며, 페이지 별로 명확한 URL이 존재해 검색엔진의 인덱싱 효율성이 높습니다.

2. **기술적 복잡도 낮음**

전통적인 방식으로 개발되기 때문에 비교적 기술적인 복잡도가 낮습니다. 서버와 클라이언트 사이의 역할 분담이 명확하고, 클라이언트 측에서의 복잡한 상태 관리나 라우팅이 필요하지 않습니다.

3. **보안 강화**

각 요청 시 서버로부터 새로운 페이지를 받아오므로 민감한 데이터가 클라이언트에 저장되지 않으며, 서버 중심의 보안 모델을 따를 수 있습니다. 이러한 이유로 보안이 중요한 서비스에서는 종종 MPA 방식을 채택합니다.

4. **단순한 페이지 관리**

독립적인 페이지 구조이기 때문에, 각 페이지는 개별적으로 개발, 유지보수, 업데이트가 가능합니다. 개발 팀이 각각의 페이지를 독립적으로 작업할 수 있어 프로젝트 관리 측면에서 유연합니다.

### MPA의 단점

1. **느린 사용자 경험**

페이지 간 전환 시마다 전체 HTML 문서와 리소스를 다시 로드해야 하기 때문에, SPA에 비해 사용자의 경험이 느려질 수 있습니다. 특히 네트워크 상태가 불안정한 경우 페이지 로딩 시간이 길어질 수 있습니다.

2. **중복된 리소스 요청**

MPA에서는 각 페이지마다 공통된 리소스(예: CSS, JavaScript)를 다시 로드해야 합니다. 이는 클라이언트와 서버 간의 불필요한 리소스 요청을 초래하여 네트워크 사용량이 증가하고, 페이지 로딩 시간을 늘릴 수 있습니다.

3. **상태 관리 어려움**

페이지가 새로고침되면서 클라이언트의 상태 정보가 초기화됩니다. 만약 사용자가 페이지를 전환할 때 특정 데이터를 유지하고 싶다면, 이를 서버 측에서 별도로 관리하거나 브라우저의 쿠키, 세션 등을 사용해야 합니다.

4. **개발 속도 저하**

각 페이지가 독립적이기 때문에, 동일한 레이아웃이나 기능을 여러 페이지에 적용할 때 중복 코드가 발생할 수 있으며, 그로 인해 개발 속도가 느려질 수 있습니다. 특히 동일한 UI 요소나 기능을 여러 페이지에서 구현해야 할 때, 반복적인 작업이 필요해질 수 있습니다.

MPA 방식의 웹 애플리케이션을 개발하는 데 도움이 되는 공식 문서들을 소개합니다:

2. [MDN Web Docs - HTML Basics](https://developer.mozilla.org/en-US/docs/Learn/Getting_started_with_the_web/HTML_basics)

MPA 구조의 웹 애플리케이션을 구성하는 기본적인 HTML에 대해 설명하는 문서입니다. 각 페이지를 독립적으로 구성하기 위한 HTML 사용 방법에 대해 배울 수 있습니다.

3. [MDN Web Docs - Web performance](https://developer.mozilla.org/en-US/docs/Learn/Performance)

MPA의 단점 중 하나인 느린 페이지 로딩을 해결하기 위한 웹 성능 최적화 가이드입니다. 웹 애플리케이션의 성능을 개선할 수 있는 다양한 기술을 소개합니다.

4. [Express.js 공식 문서](https://expressjs.com/)

MPA 웹 애플리케이션을 구축할 때 자주 사용하는 Node.js 기반 웹 프레임워크 Express.js의 공식 문서입니다. Express.js를 사용하여 서버 측에서 MPA 구조를 쉽게 구현할 수 있습니다.

## 싱글 페이지 애플리케이션 (SPA)

### 개요

싱글 페이지 애플리케이션(SPA)은 하나의 HTML 페이지를 로드하고 사용자가 앱과 상호작용할 때 동적으로 콘텐츠를 업데이트하는 웹 애플리케이션입니다. SPA는 주로 클라이언트 사이드에서 JavaScript를 사용하여 콘텐츠를 렌더링합니다.

### 장점

- **향상된 사용자 경험**: 페이지의 일부만 업데이트되므로 빠른 탐색을 제공하여 원활한 경험을 제공합니다.
- **서버 부하 감소**: 서버와 클라이언트 간에 전송되는 데이터가 적어 서버 부하가 줄어듭니다.
- **풍부한 상호작용성**: SPA는 더 동적이고 상호작용적인 사용자 인터페이스를 제공합니다.

### 단점

- **SEO 문제**: SPA는 동적으로 콘텐츠를 렌더링하기 때문에 검색 엔진 최적화(SEO)가 어렵습니다.
- **초기 로드 시간**: 애플리케이션 전체를 다운로드해야 하므로 초기 로드 시간이 길어질 수 있습니다.
- **JavaScript 의존성**: 사용자가 JavaScript를 비활성화한 경우 문제가 발생할 수 있습니다.

### 공식 문서

[React SPA 공식 문서](https://reactjs.org/docs/create-a-new-react-app.html)

## 클라이언트 사이드 렌더링 (CSR)

### 개요

클라이언트 사이드 렌더링(CSR)은 브라우저에서 직접 웹 페이지를 렌더링하는 방식입니다. 서버는 기본 HTML 문서를 보내고 클라이언트가 JavaScript를 사용하여 콘텐츠를 채웁니다.

### 장점

- **향상된 사용자 경험**: SPA와 마찬가지로 CSR은 더 반응적이고 상호작용적인 사용자 경험을 제공합니다.
- **효율적인 업데이트**: 전체 페이지를 다시 로드하지 않고 동적으로 업데이트됩니다.
- **서버 부하 감소**: 렌더링이 클라이언트에서 발생하므로 서버의 처리 부담이 줄어듭니다.

### 단점

- **SEO 문제**: SPA와 마찬가지로 CSR은 동적으로 렌더링된 콘텐츠를 검색 엔진이 인덱싱하기 어려울 수 있습니다.
- **JavaScript 의존성**: 애플리케이션이 JavaScript에 의존하므로 JavaScript가 비활성화되거나 로드에 실패할 경우 문제가 발생할 수 있습니다.
- **초기 로드 시간**: 클라이언트가 페이지를 렌더링하기 위해 JavaScript를 다운로드하고 실행해야 하므로 초기 로드가 느릴 수 있습니다.

### 공식 문서

[React CSR 공식 문서](https://reactjs.org/docs/react-dom.html)

## 서버 사이드 렌더링 (SSR)

### 개요

서버 사이드 렌더링(SSR)은 웹 페이지를 클라이언트에 보내기 전에 서버에서 렌더링하는 방식입니다. 이 접근 방식은 각 요청에 대해 서버에서 전체 HTML 페이지를 생성하여 클라이언트에 보냅니다.

### 장점

- **SEO 친화적**: 콘텐츠가 서버에서 렌더링되므로 검색 엔진에 완전히 노출되어 SEO가 개선됩니다.
- **빠른 초기 로드**: HTML이 서버에서 완전히 렌더링되므로 초기 페이지 로드가 빠릅니다.
- **저성능 장치에서의 성능**: 렌더링 부담을 클라이언트에서 서버로 오프로드하므로 제한된 처리 능력을 가진 장치에서 성능이 향상됩니다.

### 단점

- **서버 부하 증가**: 각 요청에 대해 서버에서 HTML을 렌더링해야 하므로 서버 부하가 증가할 수 있습니다.
- **느린 탐색**: 전체 페이지를 다시 로드해야 하므로 탐색 속도가 SPA보다 느릴 수 있습니다.
- **복잡성 증가**: SSR을 구현하려면 추가 인프라와 더 복잡한 설정이 필요할 수 있습니다.

## 결론

각 렌더링 전략(SPA, CSR, SSR)은 고유한 강점과 약점을 가지고 있습니다. 선택은 프로젝트의 특정 요구 사항, 예를 들어 SEO 필요성, 성능 고려 사항 및 원하는 사용자 경험에 따라 달라집니다. 이러한 차이점을 이해하면 개발자는 애플리케이션에 적합한 접근 방식을 선택할 수 있습니다.
