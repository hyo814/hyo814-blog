# 웹 성능을 높이는 방법 - 브라우저 요청 줄이기

10월 17일 프론트엔드 문제입니다!

​

웹 애플리케이션의 성능 최적화를 위해 브라우저의 요청 개수를 줄이는 방법들과, 이러한 방법이 로딩 성능에 어떤 영향을 미치는지 설명해주세요.

​

질문의도 입니다!

​

브라우저 요청 개수를 줄이는 다양한 최적화 기법과, 이러한 기법들이 로딩 성능과 유지보수성에 미치는 영향을 이해하고 있는지를 평가합니다. HTTP/2와 같은 최신 기술을 활용하는 방법과, 각 전략의 장단점을 균형 있게 고려하여 실무에서 적절한 방법을 선택할 수 있는 능력을 확인합니다.

---

### 모범 답안

웹 애플리케이션의 성능 최적화를 위해 브라우저의 요청 개수를 줄이는 다양한 방법들이 있으며, 각 방법은 성능과 유지보수성에 긍정적인 영향을 미칩니다.

### 1. **파일 병합 (Concatenation)**

여러 CSS 또는 JavaScript 파일을 하나로 병합하여 서버로의 요청 개수를 줄일 수 있습니다. 이는 네트워크 연결 설정에 필요한 시간을 절감하고, HTTP/1.1 환경에서 성능을 향상시킵니다. 그러나 파일이 커질수록 최초 로딩 시간이 길어질 수 있다는 단점이 있어, 콘텐츠의 우선순위에 맞게 병합하는 것이 중요합니다.

### 2. **파일 압축 (Compression)**

Gzip 또는 Brotli와 같은 압축 방식을 통해 파일 크기를 줄여 전송 시간을 단축할 수 있습니다. 이 방식은 HTTP/1.1과 HTTP/2 환경 모두에서 적용 가능합니다. 압축된 파일은 네트워크 대역폭을 절약하고 전송 시간을 줄여주지만, 브라우저가 압축 해제에 필요한 시간을 추가로 소요할 수 있습니다.

### 3. **이미지 스프라이트 (Image Sprite)**

여러 개의 작은 이미지를 하나의 큰 이미지로 병합하여 서버에 대한 요청 횟수를 줄입니다. CSS를 통해 필요한 부분만을 보여주는 방식입니다. 이는 이미지 리소스의 요청 개수를 획기적으로 줄일 수 있지만, 스프라이트 이미지 파일이 커질 경우 첫 로딩 시간이 길어질 수 있습니다.

### 4. **Lazy Loading**

이미지나 비디오와 같은 비필수적인 리소스를 초기 페이지 로딩 시 불러오지 않고, 사용자가 실제로 해당 콘텐츠에 접근할 때 로드하는 방식입니다. 이는 페이지의 초기 로딩 성능을 크게 향상시키며, 특히 모바일 환경에서 유용합니다. 다만, 스크롤 위치나 사용자 인터랙션에 따라 리소스 로딩이 늦어져 사용자 경험에 부정적인 영향을 줄 수 있습니다.

### 5. **HTTP/2의 멀티플렉싱 (Multiplexing)**

HTTP/2를 사용하면 하나의 TCP 연결에서 다수의 요청을 동시에 처리할 수 있어, 병렬 요청이 가능해집니다. 이를 통해 연결 설정 시간을 절약할 수 있고, 서버와의 통신 효율성을 높일 수 있습니다. 그러나 HTTP/2가 지원되지 않는 브라우저나 네트워크 환경에서는 해당 기법을 사용할 수 없다는 제한점이 있습니다.

### 6. **캐싱 (Caching)**

캐싱은 서버에서 제공하는 리소스를 브라우저에 저장하여, 이후 동일한 리소스 요청 시 다시 서버로 요청하지 않고 로컬에서 가져오도록 합니다. 브라우저 캐시, CDN 캐시 등을 활용할 수 있으며, 이를 통해 네트워크 요청 횟수를 줄이고 페이지 로딩 속도를 크게 향상시킬 수 있습니다.

### 7. **코드 스플리팅 (Code Splitting)**

JavaScript 번들 크기를 줄이기 위해 필요한 코드만을 로드하는 기법입니다. 예를 들어, `React.lazy`를 사용하여 필요한 컴포넌트를 동적으로 불러오는 방식으로 초기 로딩 시간을 단축할 수 있습니다. 그러나 코드 스플리팅이 과도하면 런타임에 새로운 스크립트를 로드하는데 시간이 걸려 사용자 경험에 악영향을 줄 수 있습니다.

---

### 로딩 성능에 미치는 영향

- **브라우저 요청 개수를 줄이면 초기 로딩 성능이 향상**됩니다. 네트워크 연결 설정과 요청-응답 과정에서 소모되는 시간을 절약하고, 중요한 리소스가 먼저 로드되도록 우선순위를 조정할 수 있습니다.
- **캐싱과 파일 병합, 스프라이트 기법 등은 반복적인 요청을 줄여** 지속적인 사용자 경험을 개선합니다.
- HTTP/2와 같은 최신 프로토콜은 **연결 수를 줄이고 멀티플렉싱을 통해 대기 시간을 줄이며** 서버 부하를 분산시키는 효과를 기대할 수 있습니다.

이러한 최적화 방법을 적절히 조합하여 사용하면 웹 애플리케이션의 성능을 크게 향상시킬 수 있으며, 실무에서는 프로젝트의 요구 사항에 맞춰 각 방법의 장단점을 고려하여 선택해야 합니다.

---

웹 페이지 로딩 시간이 느리면 사용자는 이탈할 가능성이 높습니다. 이러한 문제를 해결하기 위한 한 가지 방법은 브라우저의 요청 수를 줄이는 것입니다. 웹 애플리케이션의 성능을 최적화하기 위해 다양한 전략이 존재하며, 여기서는 대표적인 방법들을 소개하고, 각 방법이 실제 로딩 성능에 미치는 영향을 분석해 보겠습니다.

### 1. **CSS와 JS 파일 병합**

일반적으로 웹사이트는 여러 개의 CSS와 JavaScript 파일을 불러옵니다. 이 파일들을 하나로 병합하여 서버로의 요청 수를 줄이면, 브라우저가 네트워크 연결을 여러 번 설정할 필요 없이 한 번의 요청으로 필요한 리소스를 모두 받을 수 있습니다. 이를 통해 초기 로딩 속도를 단축할 수 있지만, 병합된 파일이 커질 경우 성능에 악영향을 미칠 수 있어 주의가 필요합니다.

### 2. **파일 압축 (Gzip, Brotli 등)**

CSS와 JS 파일의 크기를 줄이는 또 다른 방법은 압축입니다. Gzip이나 Brotli와 같은 압축 알고리즘을 적용하면, 파일 크기를 크게 줄일 수 있고, 전송 속도가 빨라집니다. 이를 통해 브라우저가 더 적은 데이터를 다운로드하고, 사용자에게 더 빠른 응답을 제공할 수 있습니다. 다만, 파일 압축 해제 시간이 추가로 소요될 수 있습니다.

### 3. **Lazy Loading (지연 로딩)**

이미지, 비디오 같은 대용량 리소스를 즉시 로드하는 대신, 사용자가 해당 콘텐츠에 접근할 때 로드되도록 설정하는 방식입니다. 초기 로딩 시 불필요한 리소스는 생략하고, 실제로 화면에 나타날 때만 로드함으로써 페이지 로딩 성능을 개선할 수 있습니다. 그러나 스크롤 속도가 빠른 환경에서는 늦게 로드되는 리소스가 사용자 경험에 영향을 줄 수 있어 적절한 구현이 필요합니다.

### 4. **이미지 스프라이트**

여러 개의 작은 이미지를 하나의 큰 이미지로 병합한 후, CSS를 통해 각각의 이미지를 적절한 위치에 표시하는 방법입니다. 이를 통해 이미지 파일 요청 횟수를 크게 줄일 수 있지만, 이미지 파일 크기가 커지면 초기 로딩 속도가 느려질 수 있습니다.

### 5. **HTTP/2 멀티플렉싱**

HTTP/2는 하나의 연결에서 여러 요청을 동시에 처리할 수 있는 멀티플렉싱 기능을 지원합니다. 이로 인해 브라우저가 한 번에 여러 파일을 불러올 수 있으며, 이전 HTTP/1.1에 비해 더 적은 연결로 효율적인 네트워크 통신이 가능합니다. 하지만 HTTP/2를 지원하지 않는 네트워크 환경에서는 이 기법을 사용할 수 없다는 한계도 존재합니다.

### 6. **캐싱 최적화**

캐싱을 통해 반복적인 리소스 요청을 줄일 수 있습니다. 서버에서 제공한 파일이 변경되지 않았다면, 브라우저는 이전에 캐시된 파일을 사용하여 빠르게 페이지를 로드할 수 있습니다. 브라우저 캐싱, CDN 캐싱 등을 통해 캐싱 정책을 잘 설정하면 네트워크 요청을 최소화하고 성능을 극대화할 수 있습니다.

### 7. **코드 스플리팅**

JavaScript 파일의 크기를 줄이기 위해 코드 스플리팅 기법을 활용할 수 있습니다. 즉, 사용자가 현재 페이지에서 필요로 하는 코드만 불러오고, 나머지 코드는 나중에 로드하는 방식입니다. 이를 통해 초기 로딩 시간을 줄일 수 있으며, 특히 SPA(싱글 페이지 애플리케이션)에서 효과적입니다.
